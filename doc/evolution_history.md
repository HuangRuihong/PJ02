# group ledger - 系統演化與技術挑戰全紀錄 (Evolution & Scenarios)

本文件詳實記錄了 **Split-it-Smart (Group Ledger)** 從一個簡單的記帳工具，演化為一套支援「多人連網、即時同步、債務抵銷」的完整會計系統的過程中所面臨的關鍵技術場景與決策。

---

## 🚀 階段一：本地離線架構 (Local-Only Implementation)
### 場景描述：
最初系統設計為「隱私優先」，數據僅保存在本地 SQLite。
### 挑戰與對策：
*   **挑戰**：如何在本地環境確保分帳時不會出現「1 元誤差」？
*   **突破**：實作了 **「基數 + 餘數遞增」** 演算法。如果 100 元三人分，前一人分 34 元，後兩人分 33 元，確保總金額永不失真。

---

## ⚖️ 階段二：債務生命週期狀態機 (Transaction State Machine)
### 場景描述：
在多人群組中，單方記帳可能存在爭議。
### 挑戰與對策：
*   **挑戰**：如何防止有人虛報帳目？
*   **突破**：引入 **「一票否決制 (Veto Power)」**：
    - 交易預設為 `PENDING` (待確認)。
    - 若有人點擊 `REJECTED` (有誤)，整筆交易即刻鎖定。
    - 唯有全員點擊 `CONFIRMED` 後，債務才會正式納入結算。

---

## 🌩️ 階段三：多人連網模式轉型 (Multi-user Sync Evolution)
### 場景描述：
當專案逐漸成熟，使用者提出了「跨裝置共同記帳」的強烈需求。
### 挑戰與對策：
*   **挑戰**：在不重寫 10,000+ 行前端 UI 代碼的前提下，如何將原本調用本地 SQLite 的系統轉向連接外部伺服器？
*   **突破**：採用 **「代理人模式 (Proxy/Facade Pattern)」**：
    - 實作 `NetworkDebtSystem` 類別，其介面與原有 `DebtSystem` 完全一致。
    - 使用 **FastAPI** 封裝後端業務邏輯。
    - 前端 UI 只要換插這個「網路插件」，就能在無感的情況下實現連網。

---

## 💻 階段四：Windows CMD 編碼與相容性 (The Encoding Battle)
### 場景描述：
連網模式需要啟動伺服器與腳本，在 Traditional Chinese Windows (CP950) 下遇到了嚴重的解析問題。
### 挑戰與對策：
*   **挑戰**：`.bat` 批次檔亂碼導致路徑無法解析、Emoji (`✅, ❌`) 導致診斷腳本崩潰。
*   **突破**：
    - 將所有系統內部日誌輸出與批次檔指令回饋改為 **「純英文 (ASCII Only)」**。
    - 在 Python 腳本層強制偵測 `sys.platform` 並適時注入 `UTF-8` 編碼封裝器。
    - 提供全英文的「一鍵啟動 (One-click)」環境。

---

## 🛠️ 階段五：全功能自動化診斷 (Diagnostic Mastery)
### 場景描述：
架構變得複雜（GUI + Proxy + API + Engine + DB），手動測試極其耗時且易出錯。
### 挑戰與對策：
*   **挑戰**：如何確保代碼修改後，網路同步與結算演算法依然正確？
*   **突破**：開發了 **`online_diagnostic_suite.py`** 診斷系統：
    - 自動模擬兩位虛擬使用者 (NO_1, NO_2)。
    - 全自動跑完「開群 -> 加入 -> 記帳 -> 異步確認 -> 智慧抵銷」的 100% 完整路徑。
    - 點擊 `run_full_test_online.bat` 即可在 10 秒內獲取健康報告。

---

## 📈 總結演化價值 (Evolutionary Value)
這套系統從最初的「數位筆記本」演化為具備「同步大腦」的會計軟體。其技術深度體現在：
1.  **高度擴充性**：證明了本地優先 (Local-first) 架構具備平滑遷移至雲端的能力。
2.  **強健性**：克服了跨平台編碼、網路延遲與並發確認的複雜情境。
3.  **精準度**：始終維持 0$ 誤差的結算與智慧化的債務路徑優化。
